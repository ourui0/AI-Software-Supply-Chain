## 提示注入

一般来说，提示词由两部分组成：指令提示和数据提示。指令可以来自用户、应用提供商，或两者结合，而数据则可以来自外部来源，如电子邮件、网页或 API 调用返回的内容。许多研究表明，集成 LLM 的应用容易受到即时注入攻击的侵害。模型执行攻击者选择的任务，而非预期任务。

提示注入防御现在主要分为三个方向，基于检测的防御，预防性防御，基于归因的辩护。

预防性防御是旨在通过训练或微调大语言模型，使其能够抵御恶意提示注入的同时，保证其实用性

基于归因的辩护旨在追溯到提示中导致LLM输出的恶意指令，一般用于取证分析

基于检测的防御旨在检测数据是否被污染

针对基于检测的防御模式，阅读了论文《**PIShield: Detecting Prompt Injection Attacks via Intrinsic LLM Features**》

## **PIShield: Detecting Prompt Injection Attacks via Intrinsic LLM Features**

LLM 采用的是 **Causal Attention（因果注意力机制）**。这意味着在输入序列中，**最后一个 Token** 的计算过程实际上“扫描”了前面所有的信息。

因此，最后一个 Token 的残差流向量（Residual Stream）就像是一份**全文总结报告**。PIShield 发现，如果输入里藏了恶意指令，这份“报告”在某些维度上会表现出特定的“紧张”或“异样”，且这种异样是线性可分的。

 ![image-20260316144141564](C:\Users\34021\AppData\Roaming\Typora\typora-user-images\image-20260316144141564.png)

因此**PIShield**训练了一个线性分类器，对最后一个token的残余向量流进行二元分类，检测污染数据。

针对上述方法，有一定的ID

## 与《**Killing Two Birds with One Stone: Malicious Package Detection in NPM and PyPI using a Single Model of Malicious Behavior Sequence**》的结合

发现PIShield的方法是一个静态的检测，只分析最后一个token，因此可能会漏掉一些在前面token就出现的恶意攻击

因此我们联想到上一篇论文提出的动态分析方法，给每个token进行行为ID的划分，依据行为的时序性（1. 绕过防御 -> 2. 提升权限 -> 3. 执行敏感操作）对行为ID进行线性排序，最后利用训练的模型来判断是否具有恶意行为。

但是这其中存在些许问题

+ 行为ID的划分很难处理
+ 对于每个token都分析，虽然时间开销并不会增大太多，但是内存开销会增大很多

因此还有一个方案并不针对全部token，而是针对关键token，但是关键token的寻找又是一个问题
