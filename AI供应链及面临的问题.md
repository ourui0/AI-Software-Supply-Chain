# An Exploratory Study
## 论文之前
### 什么叫AI软件供应链
AI软件供应链是传统软件供应链在人工智能时代下的演进和扩展，它在核心构成、风险焦点和管理方式上都呈现出显著的新特点。

| 对比维度 | 传统软件供应链 | AI软件供应链 |
| :--- | :--- | :--- |
| **核心构成** | 主要关注**代码组件**（如开源库、框架）及其依赖关系 | 在代码基础上，扩展至**数据、算法模型**（ML模型）及其生命周期管理 |
| **核心资产** | 源代码、可执行文件 | **数据**（训练集）和**预训练模型** |
| **关键文档** | 软件物料清单（SBOM），明确代码组件 | 人工智能物料清单（AIBOM）或机器学习物料清单（MLBOM），追溯数据来源、模型版本等 |
| **安全风险焦点** | 代码漏洞、开源组件风险、恶意代码注入 | **数据投毒**、**模型篡改**、对抗性攻击 |
| **典型新威胁** | 依赖漏洞（如Log4j）、供应链投毒（如SolarWinds事件） | 训练数据污染、AI生成恶意包（仿冒库）、模型后门 |
| **运营重心** | CI/CD管道安全、漏洞扫描与修复 | **安全的MLOps管道**、数据与模型的版本管理、模型验证与监控 |
| **独特合规挑战** | 开源许可证合规、知识产权风险 | 算法公平性/可解释性、数据隐私法规（如GDPR）、模型偏见 |

### 应对AI软件供应链的新挑战
面对AI软件供应链的复杂性和新风险，企业和开发者需要采取新的管理策略：
-   **强化数据与模型治理**：建立覆盖数据和模型全生命周期的安全管控，包括数据来源验证、清洗过程安全、模型版本控制和上线前安全测试。
-   **拥抱AIBOM/MLBOM**：像管理代码依赖一样管理AI组件，通过AIBOM实现数据、模型及其依赖关系的透明化和可追溯。
-   **构建安全的MLOps体系**：将安全实践深度集成到MLOps流程中，确保模型开发、训练、部署和更新的自动化管道安全。
-   **加强针对性防御**：关注并防御数据投毒、模型窃取、对抗性样本等新型攻击。在供应链安全管理中，需要特别关注**数据的真实性和质量**，以及**模型的完整性和鲁棒性**。


### 🔍 深入理解AI软件供应链的三大关键

除了表格中的对比，要更深入地理解AI软件供应链，还需关注以下三个核心层面，它们共同构成了AI软件供应链的独特性和复杂性。

*   **数据层：供应链的基础与风险源**。数据是AI的“新弹药”，其质量与安全直接决定模型的可靠性。与传统软件不同，AI供应链的起点往往是庞大的训练数据集。此层面面临**训练数据投毒**（恶意污染数据导致模型行为异常）、**数据隐私泄露**（处理敏感信息时违规）以及**数据来源合规性**（如使用受版权保护的数据训练模型）等独特风险。

*   **模型层：核心资产与新型攻击目标**。模型是AI软件的价值核心，也成为被攻击的新对象。模型文件本身可能包含来自训练数据的敏感信息，存在泄露风险。同时，针对模型的**对抗性攻击**（通过特殊输入误导模型）和**模型窃取**（通过API查询复制模型功能）是新型威胁。模型作为一种特殊组件，其依赖关系（如基础模型和微调模型）的合规性也极为复杂。

*   **工具与流程层：AI赋能的开发与潜在漏洞**。AI代码生成工具（如GitHub Copilot）的普及，在提升效率的同时也引入了新风险。这些工具可能产生包含不存在的“幻觉”依赖项的代码，或引入训练数据中的已知漏洞。这要求开发环节必须加强人工审查和安全测试。同时，支撑AI开发的MLOps流程，其安全性也至关重要，需防范在持续集成/持续部署（CI/CD）管道中被注入恶意代码或模型。

### 🛡️ 应对新挑战的管理策略

面对AI软件供应链的复杂性和新风险，企业和开发者需要在传统软件供应链安全管理的基础上，重点关注和加强以下几个方面：

-   **强化数据与模型治理**：建立覆盖数据和模型全生命周期的安全管控体系。这包括数据来源验证、数据清洗过程的安全保障、模型版本的严格控制以及模型上线前的全面安全测试。
-   **拥抱AIBOM/MLBOM**：像管理代码依赖一样管理AI组件。通过建立和维护**人工智能物料清单（AIBOM）**，实现对所有数据、模型及其依赖关系的透明化和可追溯性，这是有效管理的基础。
-   **构建安全的MLOps体系**：将安全实践深度集成到机器学习运维（MLOps）的全流程中。确保从模型开发、训练、部署到监控更新的自动化管道每个环节都是安全的，特别是保护训练环境和模型仓库免受未授权访问。
-   **加强针对性防御**：关注并防御数据投毒、模型窃取、对抗性样本等新型攻击。在供应链安全管理中，需要特别关注**数据的真实性和质量**，以及**模型的完整性和鲁棒性**。

### 1. 📈 AI 融入软件供应链，安全实践正在重新定义
这篇文章指出，AI与企业软件开发关系的日益紧密，使得AI本身已成为软件供应链的重要组成部分，但这同时也增加了供应链的复杂性和新的安全威胁。

*   **新风险与挑战**：AI的引入形成了新的安全风险生态，主要包括：
    *   **生成式AI毒化**：恶意行为者向开源训练数据中注入欺骗性数据或后门。
    *   **AI生成的拼写欺诈**：AI自动创建与合法开源包名称相似的虚假包，欺骗开发者安装恶意软件。
    *   **分发恶意软件的虚假项目**：AI生成看似真实的代码库，其中包含可绕过传统安全检查的恶意软件。
    *   **AI增强的网络钓鱼**：攻击者使用AI制作针对供应链利益相关者的可信网络钓鱼邮件。
*   **AI的三重角色与应对**：
    *   **AI作为开发者**：AI代码助手可能引入安全漏洞，因其训练数据可能包含已知漏洞。
    *   **AI作为模型**：机器学习模型（如Hugging Face平台上的模型）本身可能成为被攻击的目标。
    *   **AI作为攻击者**：威胁行为者利用AI技术增强其攻击能力。
*   **关键管理策略**：文章建议通过**机器学习物料清单（MLBOMs）或人工智能物料清单（AIBOMs）** 来跟踪AI模型的组件和依赖关系，并确保**CI/CD管道**的安全。

## 🔍 真实的模型窃取事件：谷歌提取OpenAI模型参数
2024年，谷歌的研究团队进行了一项引起广泛关注的实验，他们通过一种称为“模型窃取”的技术，成功地从OpenAI的ChatGPT模型中提取了关键参数。

攻击手法：研究团队利用OpenAI提供的API接口，向ChatGPT的Ada和Babbage基础模型发送了不到2000次经过精心设计的查询。通过分析模型对这些查询产生的输出，他们成功推断出了模型内部投影矩阵的结构，甚至破解了其内部隐藏维度（分别为1024和2048）这一关键信息。

影响与特点：这项攻击最令人惊讶之处在于其极低的成本和高效性。整个提取过程仅花费了约20美元（约合150元人民币），并且该方法据称同样适用于更强大的GPT-3.5和GPT-4模型。这清晰地表明，即使是业界领先的闭源商业模型，其核心信息也可能通过此类旁路攻击被窃取事件曝光后，OpenAI已迅速修改了其API接口以防止此类攻击被复现。

后续发展：需要说明的是，谷歌此举是经OpenAI同意的正规安全研究，旨在揭示风险，并在研究后删除了所有相关数据。但这一实验无疑为所有AI公司敲响了警钟，凸显了模型接口可能存在的安全风险。


💻 事件背后的攻击原理
“Slopsquatting”攻击之所以危险，在于它精准地利用了AI编码工作流中的信任环节。其攻击链条非常清晰 ：

AI产生幻觉：当开发者使用AI编码助手时，AI可能会在生成的代码中推荐一个看起来完全合理（例如 huggingface-cli），但实际在官方软件仓库（如PyPI）中并不存在的包名。研究发现，开源模型生成的代码中，高达约22%的依赖项可能是虚构的 。

攻击者抢注投毒：攻击者会持续监控AI生成的代码，发现这些被频繁“幻想”出来的包名后，迅速在公共软件仓库中将其注册，并在包内植入恶意代码。

开发者中招：如果开发者未加验证，直接运行AI生成的安装命令（如 pip install huggingface-cli），就会从攻击者控制的服务器下载并执行恶意软件包。恶意代码可能借此窃取敏感数据、植入后门，甚至进一步渗透到整个开发和生产环境 。
